# Exercise 1 · Appendix

The supporting data for the audit, provided so the analysis can be inspected, reproduced, or extended rather than taken on trust.

## The data set

The audit is a single CSV with one row per live URL. It is the source of every number and every evidence table in this exercise.

<DownloadLink href="/artifacts/documentation-audit.csv">Download the full audit (CSV, 420 rows)</DownloadLink>

### Schema

| Column | Type | Meaning |
|--------|------|---------|
| `url` | string | The current live URL, the unit of analysis |
| `page_title` | string | The page's own title |
| `proposed_l1`-`proposed_l4` | string | Proposed placement, up to four levels |
| `url_flag` | enum-ish | Action: `DUPLICATE …`, `REDIRECT …`, `MOVE …`, `KEEP …`, or empty (implicit keep) |
| `writer_placement_rule` | string | One-line rationale, doubles as a reusable content-model rule |

## A note on versioning

Two versions of the audit exist. An **earlier draft** kept `Deploy`, `Le Chat`, and `Evaluate & Ship` as separate top-level sections and let `Reference` bloat to 176 pages. The **canonical version** used throughout this submission collapses those into the six-section model and holds Reference to 49.

The difference is itself evidence for [R5](/exercise-1/recommendations#r5): without a strict "look up vs. learn" admission test, Reference re-bloats. Keeping both versions documents the reasoning, not just the conclusion, the earlier draft shows the failure mode the final model was designed to avoid.

## Reproducibility

The summary statistics on the [Audit Findings](/exercise-1/audit-findings) and Executive Summary pages are computed directly from this CSV at build time, the counts you read are derived from the data, not typed by hand. Re-running the parse against an updated export regenerates them.
