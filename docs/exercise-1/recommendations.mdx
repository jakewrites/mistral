# Recommendations

Three recommendations follow. Each uses the same template (Observation, Evidence, Recommendation, Expected impact, Effort, Priority) so they can be compared on equal terms. Every one cites the finding in [The Audit](/exercise-1/audit-findings) that motivated it. The [Sequencing](#sequencing) section below turns them into a phased plan, [Considerations](#considerations) raises a cross-cutting strategy question the recommendations surface (documenting self-hosted products), and [After it Ships](/exercise-1/after-it-ships) names the post-ship signal for each and the operating model behind them. The calls the proposal leaves open are named honestly on [Open Questions](/exercise-1/open-questions).

:::note The three at a glance
**R1** re-cuts the top nav into six intent-based sections and dissolves the audience-named `Developers` section. **R2** restricts Reference to true look-ups and gathers the scattered ones into it. **R3** fixes the machine-readable fundamentals (`robots.txt`, content negotiation, a working `llms.txt`). R1 is the umbrella; R2 is its guardrail against drift, and R3 is the same canonical-URL discipline applied to the agent reader.
:::

<Recommendation
  id="R1"
  title="Reorganise the top navigation into six intent-based sections"
  priority="P1"
  impact="High"
  effort="Medium"
  observation="The top navigation encodes product surfaces and audiences (Products, Developers, Admin) rather than user goals, so readers must know which Mistral surface owns their task before they can navigate. Developers is the worst offender: an audience-named catch-all whose actual building work lives under Products › Studio."
  recommendation={<>Adopt six task-based top-level sections, <strong>Get Started · Models · Build · Products · Admin · Reference</strong>, and move every page to the section that matches the reader's job. Dissolve the audience-named <strong>Developers</strong> section into them (orientation to Get Started, guidance to Build, reference to Reference), name each canonical URL for the <strong>task</strong> rather than the persona or product, and 301-redirect the redundant pages. Then add what the sections deliberately do not encode: an orthogonal <strong>surface axis</strong> (managed | self-hosted), threaded through the sections as paired entry points and version-gating, not a section of its own.</>}
  expectedImpact="Navigation answers 'what do I want to do?' instead of 'which product is this, and who are you?'. This is the umbrella change the rest of the proposal builds on, and it caps top-nav cognitive load.">

  This is the architecture the rest of the proposal moves pages into. The full target tree is the [Information Architecture redesign](/appendix) embedded in the Appendix.

  | Section | Job it answers |
  |---|---|
  | Get Started | "Orient me and get me a first success" |
  | Models | "Choose a model; run it on either surface" |
  | Build | "Help me build this capability" (incl. Evaluate & Ship) |
  | Products | "Use the products" (Vibe, Studio) |
  | Admin | "Set up my organisation" |
  | Reference | "Give me the exact source of truth" |

  (Community and contributing content sits outside the six sections: the Community, Ambassadors, and Contribute pages, plus the footer's social links. Community is a channel, not a documentation task.) Each of the six is a *section*. The managed/self-hosted **surface** is a second, orthogonal axis, explained below.

  **Dissolve Developers.** [Finding 1](/exercise-1/audit-findings) has the evidence. An audience-named container has no admission rule, so nothing answers "does this page belong here?" Give every Developers page its one canonical home: orientation and quickstart links to Get Started, task guidance to Build, and reference material (changelogs, SDKs, the error glossary, migration guides) to Reference. The developer persona becomes a landing-page entry point, not a section. Removing the section stops the drift at its source, because an audience section can never enforce a content model, so it will always compete with the task sections for the same content.

  **Task-named canonical URLs.** Quickstart URLs today carry persona and product segments (developer, vibe-work), and product names churn: the same quickstarts have already lived under `le-chat` paths. Name each canonical URL for the task, and 301-redirect the redundant pages:

  | Resolves today | Should |
  |---|---|
  | `/developers` | Redirect into the task sections as the section dissolves |
  | `/products` | Redirect to the Get Started platform-overview orientation page |
  | `/models/overview` | Redirect to `/models`, which does the same job |
  | `/resources/cookbooks/quickstart` | Redirect to the first-api-request quickstart it duplicates |

  Beyond those four, the site serves the *same* page at two URLs through an optional `/en/` locale prefix: `docs.mistral.ai/en/studio-api/search-toolkit/changelog` and the same path without `/en/` both return `200`, rather than one `301`-ing to the other (verified live). Spot-checks across unrelated sections show the same double-`200`, so it is site-wide: every page has a duplicate, and neither form carries a `rel="canonical"` tag (verified 6 July 2026). By contrast `/fr/` is a genuine locale the sitemap lists, which makes the bare-vs-`/en/` duplication a pure config defect: choose one canonical form, `301` the other, add `rel="canonical"`. It compounds [R3](#r3)'s machine-readability point, since bots split authority on duplicate URLs exactly as humans do.

  **On Products.** The section keeps the live site's name because it already does the right job and readers know it. It is the counterpart to Build (building *with* the API versus *using* the products), and the one section whose reader may not be a developer at all, so its pages stay UI-first, name screens rather than endpoints, and link to Build for anything that teaches the API. Studio gets the same admission rule as Models: describe the console (what the screens do, where API keys live) and send API-building guidance to Build. Whether a product runs as managed SaaS or self-hosted is a question for the surface axis, so the self-hosted Le Chat pages live here too, tagged self-hosted and versioned.

  **Two audiences, one IA.** The six sections are a developer's journey, but the Vibe user shares two touchpoints with them, choosing a model and writing prompts. The IA serves both from one structure through progressive disclosure, not a second site or duplicated content. The [Audit's three journeys](/exercise-1/audit-findings#three-journeys) walks how; the rule is to split by *task*, not audience, so a shared task like prompting is one progressive page while a Vibe-only task like driving Chat's UI lives only in Products.

  **The surface axis fixes Models without a new section.** Choosing a model and operating a self-hosted cluster are two *surfaces* of one subject (managed and self-hosted), not two sections: self-hosting is *where* you run it, not *what* you are doing. A self-hosted section would strand SSO-on-self-hosted in Admin and version notes in Reference, so instead the IA threads a surface control through the sections. Inside Products, a product that ships both ways gets **paired entry points** (Vibe managed, Vibe self-hosted); the reader picks once and sees only their surface. Shared material is **single-sourced**, rendered into both trees and never copied, so it cannot fork; only genuinely different pages (install, configure, upgrade) get variants. Self-hosted trees are **version-gated** while managed stays unversioned ([Le Chat Enterprise ships both](https://mistral.ai/news/le-chat-enterprise/)). Each variant keeps its own URL, so search and agents land on the right surface (the canonical-URL discipline above), and every split page opens with a one-line surface marker linking its counterpart, since search will otherwise drop readers into the wrong tree. This is specified as a design, not yet prototyped: [Exercise 4](/exercise-4-onprem-ia) builds the self-hosted operate lifecycle and the [versioning policy](/exercise-4-onprem-ia#versioning) behind the gate. Where self-hosted content is sensitive enough to sit behind a login, [Considerations](#considerations) sets the single-sourcing and discoverability rules.

</Recommendation>

  The mixing runs in both directions on the live site:

  | Live today | What it is | Where it belongs |
  |---|---|---|
  | `/getting-started/glossary` | Pure definitions | Reference (flagged `MOVE`) |
  | `/robots` | Crawler identification spec | Reference (flagged `MOVE`) |
  | `/resources/error-glossary`, `/resources/known-limitations` | Look-up | Reference, next to the specs they explain |
  | `/resources/cookbooks/…` (125 pages) | Teaching examples | The Cookbooks utility surface, not blended with look-up content |
  | `/resources/cookbooks/quickstart` | A quickstart | Get Started (flagged `REDIRECT`) |

  Reference is where teaching content quietly accumulates, because everything technical looks like reference to someone. Without a strict admission test the section balloons. The "look up vs. learn" line is what keeps it focused. Model specs are the one look-up type that stays out of Reference, and the reason is intent, not content type: choosing a model is a primary job at an LLM company, so the specs live in Models with the decision they support. Reference keeps only the look-ups that no section owns.

</Recommendation>

  expectedImpact="Agents can discover, crawl, and cite the docs against stable canonical URLs instead of hitting 404s, and the machine index stops advertising dead links.   Verified live against `docs.mistral.ai` on 5 July 2026. The public score is **7/100** on [isitagentready.com](https://isitagentready.com/docs.mistral.ai), and the reasons that matter for a docs site are concrete and real:

  | Signal | State | Action |
  |---|---|---|
  | `robots.txt` on the docs subdomain | ❌ **404**, no crawl rules at all | **Add it**: AI-bot rules + `Sitemap:` directive |
  | Content negotiation (serve the per-page `.md`) | ❌ **404** on a plain GET | **Add it** so the Markdown `llms.txt` advertises resolves |
  | `llms.txt` (structured index) | ⚠️ present, but its advertised links **404** | **Fix the links** once negotiation works |
  | `llms-full.txt` (full-content dump) | ✅ present (~1 MB) | Keep |
  | `sitemap.xml` | ✅ present | Reference it from the new `robots.txt` |

  A structured `llms.txt` existing is not the same as it *working*: an agent that follows the per-page links it advertises hits 404s, so "we have an `llms.txt`" was the wrong thing to take comfort in. The score aggregates further checks too, and some of what it weights is speculative and not all relevant to documentation (MCP-in-robots, DNS-AID, commerce protocols like x402/ACP), but that does not rescue the grade. The fundamentals a docs site is actually judged on, crawlability and content negotiation and a machine index whose links resolve, are the ones failing.

  This is the same thesis as the rest of Exercise 1, one audience further out. The reader spectrum runs Vibe user to developer to infrastructure engineer. The agent is simply the next point on it, and it wants exactly what the humans want: one canonical home per thing and a structure it can predict. Fix the fundamentals above, ship [R1](#r1)'s canonical URLs, and agent-readiness rises with the IA work already proposed.

</Recommendation>

## Sequencing {#sequencing}

Three recommendations are only useful in the right order, by impact, effort, and dependency.

| ID | Recommendation | Impact | Effort | Priority |
|----|----------------|--------|--------|----------|
| [R1](#r1) | Reorganise the top nav into six sections; dissolve Developers | High | Medium | **P1** |
| [R2](#r2) | Make Reference the exact source of truth | Medium | Medium | **P2** |
| [R3](#r3) | Make the docs agent-ready (robots.txt, `.md` negotiation) | High | Low | **P2** |

R1 is one recommendation but two shipping phases, because its config-only parts can land long before the restructure:

- **Phase 1, stop the bleeding (days):** ship R1's pure-config parts (the four redirects and the task-naming rule) alongside the **R3** infrastructure fixes (a docs `robots.txt`, working `.md` endpoints). No structural agreement needed, a visible win that de-risks the rest.
- **Phase 2, restructure (weeks):** execute the rest of R1, the six-section re-cut and the dissolution of Developers. Page moves and redirects, with no rewrites, shipped together so readers get one coherent change.
- **Phase 3, refine (ongoing):** apply **R2** as the guardrail that keeps Reference from drifting back to a catch-all.

Each phase should ship with a named success signal, captured against a pre-launch baseline, so "the restructure felt better" becomes a number. [After it Ships](/exercise-1/after-it-ships) defines the signal for each recommendation, the one product metric worth fighting for, and the operating model that keeps Engineering, Product, and Support equally served.

## Considerations {#considerations}

A few questions sit *above* the numbered recommendations, because they shape the content model itself. The sharpest is self-hosting, and it is not hypothetical: Mistral ships **Vibe / Le Chat Enterprise in Cloud, Serverless, and Self-Hosted editions** ([announcement](https://mistral.ai/news/le-chat-enterprise/)), and its open-weight models can be self-deployed. So "how do we document a product the customer runs in *their* environment?" is a live question the IA has to answer.

### Self-hosted products: single-sourced, and public even when gated

[R1](#r1)'s surface axis handles the *public* self-hosted content (inference serving, config, upgrades), which is most of it. The harder subset is genuinely sensitive (private runbooks, customer topology, licence-gated components) and legitimately sits behind a login. Two rules hold **whether a page is public or gated**:

1. **Single-source it.** Gating decides *who sees the rendered page*, never *where the source lives*. Gated pages are authored in the same repo and pipeline as the public docs; a separate private copy drifts, and drift in deployment docs is how a customer bricks a cluster.
2. **Keep the *whether* public even when the *how* is gated.** Public docs must always state, in the open: that self-hosting exists, when to choose it, and the concrete path to the gated detail (a self-serve entitlement, or an explicit *"contact us"* route). A capability a public reader cannot discover is, to them, a capability that does not exist.

**Where this bites today.** Self-hosted Vibe lives in marketing and a help-centre "talk to our team" article, not the docs, so the option is barely discoverable and there is no public bridge to the detail. The fix: a **public stub page per self-hostable product** (what it is, when to choose it, the entitlement-or-contact path), with the runbook gated *behind* it and single-sourced *with* it.
